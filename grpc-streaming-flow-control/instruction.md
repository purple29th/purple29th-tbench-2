You're fixing an HTTP/2 multiplexer, the thing gRPC runs on the wire: one connection, many concurrent streams, with flow control and priority so no stream hogs everything. The code in /app/src/com/example/grpc/StreamMultiplexer.kt gets the frame logs wrong in a few cases, and your job is to make it right. Only change StreamMultiplexer.kt; leave Main.kt, FrameTypes.kt, and StreamState.kt alone. The verifier builds and runs it for you.

The driver reads /app/scenario.txt one operation per line. OPEN_STREAM <streamId> <weight> opens a client stream with a weight from 1 to 256. SEND_HEADERS <streamId> emits a HEADERS frame, which renders with bytes=0 and touches no window. SEND_DATA <streamId> <bytes> <endStream> queues a DATA frame; emitting it spends both the stream and the connection windows, and if endStream is true the stream becomes half-closed once those bytes have drained. WINDOW_UPDATE <streamId|0> <inc> adds credit to a stream window when the id is nonzero, or to the connection window when the id is 0. RST_STREAM <streamId> aborts a stream, so its queued data is dropped, its remaining window credit is released, and from then on the snapshot shows window=0. ACK_BYTES <streamId> <bytes> marks that many already-sent bytes as consumed and returns them to the connection window. HALF_CLOSE_LOCAL <streamId> half-closes the sender so no more DATA is queued, though the stream can still receive WINDOW_UPDATE. TICK drains queued DATA across every stream under flow control and priority. QUERY appends a state snapshot to /app/output.txt.

There are two windows. The connection window is shared by all streams, starts at 65535, drops by emitted DATA bytes, and refills from WINDOW_UPDATE on id 0 or ACK_BYTES on any stream. Each open stream also has its own window that starts at 65535, drops by that stream's emitted DATA, and refills from WINDOW_UPDATE on its id. A DATA frame spends both windows; a HEADERS frame spends neither. The connection window also keeps a reserved floor of 4096 bytes that DATA can't touch, so a TICK hands out at most (connectionWindow minus 4096) bytes total across contending streams, clamped to zero once the connection window is at or below 4096. Emitted bytes still subtract from the real connection window as usual. The floor is only a per-tick allocation cap, not a separate counter.

Stream ids carry parity, so clients use odd ids, servers use even ids, and id 0 is the control stream that accepts only WINDOW_UPDATE. A stream starts IDLE, becomes OPEN, may move to HALF_CLOSED_LOCAL, and finally reaches CLOSED; alternatively it goes straight from OPEN to CLOSED on RST_STREAM. It becomes half-closed when an endStream DATA fully drains, or right away on HALF_CLOSE_LOCAL, and while half-closed it still applies WINDOW_UPDATE credit even though it won't send more DATA. Anything aimed at a CLOSED or never-opened stream is silently ignored, and so is anything that breaks parity, exceeds the weight range, or misuses the control-stream id.

A TICK drains queued DATA. When the contending streams' total queued bytes exceed the available connection window, that window splits across them by weight. A stream contends when it is OPEN or HALF_CLOSED_LOCAL, has queued DATA, and still has stream-window credit. The bytes it gets this tick are the smallest of its queued bytes, its remaining stream-window credit, and its proportional share, which is floor(availableConnectionWindow * streamWeight / sumOfContendingWeights). Leftover bytes from that integer division go to the contending stream with the highest weight, and ties on weight go to the lowest stream id. The drain repeats within the TICK until no contending stream can make progress, meaning no queue, no stream-window credit, or no connection-window credit is left. Within a TICK, the resulting DATA frames are appended to the frames log in the order their streams were opened (the order OPEN_STREAM was issued).

A QUERY appends a snapshot in this shape:
  connection window=<n>
  streams:
    id=<n> state=<STATE> weight=<w> window=<n> queued=<n> sent=<n>
    (one line per stream)
  frames:
    <TYPE> stream=<n> bytes=<n>
    (one line per frame)
The streams part lists every stream ever opened, sorted by id ascending, and a closed stream stays listed as CLOSED with queued 0, window 0, and sent equal to the bytes it emitted before closing. The frames part is the running log of every frame successfully emitted, in emission order. HEADERS log bytes=0, DATA log the bytes actually sent, and rejected operations log nothing.
