This is a simulator of the Android Choreographer doFrame loop and the main thread Looper MessageQueue that drives each UI frame. Each DO_FRAME is one vsync, it runs four phases INPUT, ANIMATION, INSETS, TRAVERSAL in that order, drains the Looper queue between phases, and flags jank when work overruns the deadline. Fix the DoFrameScheduler file to match the behavior here, leave the main file and types files alone. The verifier builds and runs it.

The driver reads a scenario file one operation per line. SET_VSYNC_RATE sets the rate to sixty, ninety or one twenty, interval is one thousand divided by hz using integer division. Its new interval is recorded as pending and applied only at end of next DO_FRAME, so it first takes effect one frame later and that next frame keeps its previous interval and deadline. POST_FRAME registers a callback for the next frame with phase INPUT ANIMATION INSETS TRAVERSAL and repeat once or repeat. REMOVE_FRAME cancels a callback both queued for next frame and in flight. POST_INPUT queues an input event at logical time eventTime. POST_MESSAGE queues a Looper message with priority sync or async. SCHEDULE_TRAVERSALS posts a sync barrier that blocks sync messages until next frame TRAVERSAL. DO_FRAME runs one frame. DRAIN_QUEUE drains the queue outside a frame up to budget with no partial dispatch. QUERY appends a snapshot to output file.

Intervals are sixty is sixteen, ninety is eleven, one twenty is eight, start at sixty. For each DO_FRAME with vsyncTime, deadline is vsyncTime plus frameInterval. currentTime advances only by dispatched message cost and executed callback cost, phase transitions, queue checks, barrier lift and rate changes cost nothing. At start of each DO_FRAME, if currentTime is below vsyncTime it jumps to vsyncTime. Frame counter starts at one on first DO_FRAME and increments each DO_FRAME.

Each DO_FRAME runs in fixed order. It begins with input resampling where events with event time less equal vsync minus four dispatch this frame and rest defer. If any dispatched emit INPUT_BATCH count and oldest event time, if any deferred emit INPUT_DEFERRED count, no time cost. Then INPUT callbacks run in posting order each emitting FRAME_CALLBACK and adding cost to current time. Between INPUT and ANIMATION it drains async while current time less deadline, dispatching queued async first in first out, emitting MSG_DISPATCH and adding cost, stopping at deadline or when no eligible async remains. ANIMATION callbacks same, then async drain, then INSETS callbacks, then async drain. At start of TRAVERSAL emit MSG_BARRIER_LIFTED and mark barrier inactive on every frame whatever prior state, then run TRAVERSAL callbacks same way. After TRAVERSAL final drain while current time less deadline dispatching eligible messages async always eligible and sync only when barrier inactive, emitting MSG_DISPATCH. If current time ever passes deadline during frame, that frame is jank, emit JANK once at end. Work is never deferred because of jank.

Callbacks target next frame not running one and run in posting order within phase. Once is removed after running, repeat runs once per frame and re registers at end of its phase next frame queue. REMOVE_FRAME drops token from next frame queue and in flight and during a frame also scrubs any re registration a repeat would have produced.

Resampling detail is offset four, dispatch if event time less equal vsync minus four. Dispatched events coalesce into one INPUT_BATCH count and oldest time, deferred ones into one INPUT_DEFERRED count, and position payloads are not printed.

Queue has two first in first out lanes sync and async. Barrier active sync waits async dispatches, inactive both dispatch. SCHEDULE_TRAVERSALS posts a sync barrier and emits MSG_BARRIER, but if barrier already active it is a no operation with no second MSG_BARRIER, every DO_FRAME still emits MSG_BARRIER_LIFTED at TRAVERSAL start. DRAIN_QUEUE outside a frame dispatches up to budget total cost with no partial dispatch, each step dispatches eligible async first then eligible sync if barrier inactive, stopping when nothing fits, both lanes empty, or barrier blocks rest.

SET_VSYNC_RATE records a pending interval applied at the end of the next DO_FRAME, so that frame keeps its current interval and deadline and the new interval applies on the following frame; a frame in progress keeps its interval and deadline. Emit VSYNC_RATE_CHANGED from=old to=new IMMEDIATELY when the SET_VSYNC_RATE line is processed in the driver input (command-read time), NOT deferred to the later apply point. That is, the VSYNC_RATE_CHANGED event appears in the events log exactly at the point the SET occurs between DO_FRAMEs, while the deadline change is delayed one frame. Example to disambiguate: sequence DO_FRAME 0, SET_VSYNC_RATE 120, DO_FRAME 16, DO_FRAME 24 yields events MSG_BARRIER_LIFTED (frame 1), VSYNC_RATE_CHANGED from=60 to=120, MSG_BARRIER_LIFTED (frame 2 still deadline 32 using old 60 Hz interval 16), MSG_BARRIER_LIFTED (frame 3 deadline 32 using new 8 ms interval). If you emit VSYNC_RATE_CHANGED at apply time (end of next DO_FRAME) you will be one position too late and fail vsync_rate_change, adversarial_complex, gauntlet.

QUERY appends snapshot with vsync rate, frame interval, current time, then frames block with frame vsync time deadline spent and jank, then queue block with sync async barrier, then next frame callbacks, then events. Frames is cumulative in order, every phase line always present even empty, listing tokens that executed in that phase in dispatch order. Queue shows pending counts and barrier state. Next frame callbacks lists tokens registered for next DO_FRAME across phases in registration order. Events is cumulative log in emission order.
